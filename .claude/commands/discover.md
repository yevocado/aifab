# /aifab:discover — 아키텍처 디스커버리 세션

## 개요

사용자의 요구사항에서 최적의 아키텍처를 도출하는 구조화된 디스커버리 세션을 진행한다.

**핵심 원칙:** 아키텍처는 사용자 요구에서 자연스럽게 도출되어야 한다. 어떤 기술도 사전에 전제하지 않는다. 모든 프로젝트는 새로운 분석에서 시작한다.

---

## Step 1: 입력 방식 선택

사용자에게 다음 중 하나를 선택하도록 안내한다. **반드시 이 메시지를 그대로 출력한다:**

```
안녕하세요! 아키텍처 디스커버리 세션을 시작하겠습니다.

프로젝트 정보를 어떻게 제공하시겠어요?

A) 기능/프로젝트를 자연어로 설명
B) 기존 코드베이스 디렉토리 경로 제공 (분석)
C) 요구사항 문서 경로 제공

A, B, C 중 하나를 선택해주세요.
```

사용자 응답을 기다린다.

---

## Step 2: 입력 분석

선택에 따라 다음 분석을 수행한다.

### (A) 자연어 설명
- 핵심 개념, 도메인, 기능 요구사항을 추출한다
- 언급된 기술 키워드, 규모 힌트, 제약 조건을 메모한다
- 아직 불명확한 영역을 파악한다

### (B) 코드베이스 분석
- `Read`와 `Bash` 도구로 디렉토리를 스캔한다
- 다음 항목을 파악한다:
  - 현재 기술 스택 (언어, 프레임워크, 패키지 매니저)
  - 폴더 구조와 아키텍처 패턴
  - 설정 파일 (Dockerfile, docker-compose, CI/CD)
  - 테스트 전략
  - 기존 컨벤션과 패턴
- 파악된 정보를 요약한 후 질문으로 넘어간다

### (C) 요구사항 문서 파싱
- `Read` 도구로 문서를 읽는다
- 다음을 식별한다:
  - 필수(must-have) 기능
  - 명시된 기술 제약
  - 비기능 요구사항 (성능, 보안, 가용성)
  - 타임라인, 예산 언급
- 파악된 정보를 요약한 후 질문으로 넘어간다

---

## Step 3: 디스커버리 질문 (한 번에 하나씩)

**중요: 질문은 반드시 한 번에 하나씩만 한다. 답변을 받은 후 다음 질문으로 넘어간다.**

입력 분석 결과를 바탕으로 아래 8개 영역에서 맥락에 맞는 질문을 생성한다. 이미 입력에서 명확히 파악된 항목은 건너뛰되, 불명확하면 반드시 질문한다. 총 5~10개 질문을 목표로 한다.

### 필수 질문 영역

**[핵심 기능]**
> "이 시스템의 가장 중요한 기능 3가지는 무엇인가요?"

**[사용자/규모]**
> "예상 사용자 수와 트래픽 패턴은 어떻게 되나요? (예: 소규모 내부 도구 vs 수만 명의 외부 사용자)"

**[실시간 요구]**
> "실시간 기능이 필요한가요? (채팅, 라이브 업데이트, 알림 등)"

**[데이터 복잡도]**
> "데이터 모델의 복잡도는? 관계형 데이터가 많은가요, 비정형 데이터가 많은가요?"

**[AI/LLM 통합]**
> "AI나 LLM 기능을 통합할 예정인가요? 어떤 방식으로?"

**[팀 역량]**
> "개발팀의 주요 기술 경험은 무엇인가요?"

**[배포/운영]**
> "배포 환경은 어떻게 되나요? (클라우드, 온프레미스, 서버리스 등)"

**[제약]**
> "예산, 타임라인, 기술 제약이 있나요?"

### 추가 후속 질문 (답변 내용에 따라 생성)
- 실시간 기능이 있다면: 동시 접속자 수, WebSocket vs SSE vs 폴링 선호도
- AI 통합이 있다면: 스트리밍 응답 필요 여부, 모델 선택 자유도, 비용 민감도
- 대규모 트래픽이라면: 캐싱 전략, CDN 필요 여부, DB 읽기 복제본 고려 여부
- 마이크로서비스 언급 시: 팀 규모, DevOps 성숙도, 서비스 간 통신 선호 방식

---

## Step 4: 아키텍처 옵션 합성

모든 질문의 답변을 수집한 후, 답변된 요구사항을 기반으로 **2~3개의 아키텍처 옵션**을 합성한다.

**중요: 옵션은 사전에 정해진 템플릿이 아니라 수집된 답변에서 도출해야 한다.**

각 옵션에 대해 다음 형식으로 제시한다:

### 아키텍처 비교표

| 항목 | 옵션 1: [이름] | 옵션 2: [이름] | 옵션 3: [이름] (해당 시) |
|------|--------------|--------------|----------------------|
| **기술 스택** | [구체적 기술 목록] | [구체적 기술 목록] | [구체적 기술 목록] |
| **적합한 이유** | [수집된 답변과 매칭] | [수집된 답변과 매칭] | [수집된 답변과 매칭] |
| **트레이드오프** | [단점/리스크] | [단점/리스크] | [단점/리스크] |
| **예상 개발 기간** | [대략적 타임라인] | [대략적 타임라인] | [대략적 타임라인] |
| **팀 적합도** | [팀 경험 기반 평가] | [팀 경험 기반 평가] | [팀 경험 기반 평가] |

기술 스택 예시 형식 (상황에 맞게 도출):
- `FastAPI + PostgreSQL + React + Docker`
- `Next.js (풀스택) + Vercel + Supabase`
- `Go + gRPC + Kafka + React + Kubernetes`
- `Rails + Hotwire + PostgreSQL + Heroku`

**어떤 기술도 사전에 배제하거나 선호하지 않는다.**

---

## Step 5: 추천 및 선택

### 추천 발표

비교표 다음에 명확한 추천을 제시한다:

```
## 권장 아키텍처: [옵션 이름]

**추천 이유:**
[수집된 답변 항목들과 구체적으로 연결하여 설명]
- [요구사항 A] → [이 옵션이 적합한 이유]
- [요구사항 B] → [이 옵션이 적합한 이유]

**주의할 점:**
[이 옵션을 선택할 때 감수해야 하는 트레이드오프]

어떤 옵션을 선택하시겠어요? (1, 2, 3 또는 직접 입력)
```

### 선택 확인

사용자가 선택하면:
```
[옵션 이름] 아키텍처로 진행하겠습니다.

이 선택은 다음을 의미합니다:
- [구체적 의미 1]
- [구체적 의미 2]
- [구체적 의미 3]

ARCHITECTURE.md에 저장하고 워크로그를 초기화합니다.
```

---

## Step 6: 저장 및 초기화

선택 확인 후 다음 순서로 실행한다.

### 6-1. ARCHITECTURE.md 생성

현재 디렉토리에 `ARCHITECTURE.md`를 생성한다:

```markdown
# 아키텍처 결정 문서

## 프로젝트 개요
[사용자가 설명한 프로젝트 요약]

## 선택된 아키텍처
**[선택된 옵션 이름]**

## 기술 스택
[구체적 기술 목록]

## 결정 근거
[수집된 답변과 선택 이유 연결]

## 트레이드오프 (인지된 리스크)
[선택 시 감수한 트레이드오프]

## 검토된 대안
[검토했으나 선택되지 않은 옵션들과 이유]

## 결정일
[오늘 날짜]
```

### 6-2. 워크로그 초기화

다음 명령을 실행한다:
```
/aifab:worklog init [프로젝트-이름]
```

프로젝트 이름은 사용자가 설명한 프로젝트에서 추출한 영문 소문자 케밥케이스로 한다 (예: `my-saas-app`, `internal-crm-tool`).

### 6-3. WORKLOG.md 아키텍처 결정 기록

WORKLOG.md에 다음 항목을 추가한다:
```
## 아키텍처 결정
- 선택: [아키텍처 이름]
- 기술 스택: [스택 요약]
- 결정 근거: [핵심 이유 1-2줄]
```

### 6-4. 완료 안내

마지막으로 다음 메시지를 출력한다:

```
아키텍처 결정 완료.

저장된 파일:
- ARCHITECTURE.md (아키텍처 결정 문서)
- WORKLOG.md (프로젝트 워크로그 초기화됨)

다음 단계: `/aifab:plan`으로 개발 플랜을 작성하세요.
```

---

## 중요 규칙 요약

| 규칙 | 내용 |
|------|------|
| 기술 편향 없음 | Python, TypeScript, Go, Rust, Ruby 등 어떤 언어도 사전에 선호하지 않는다 |
| 질문은 하나씩 | 여러 질문을 한꺼번에 던지지 않는다. 답변 후 다음 질문으로 넘어간다 |
| 답변에서 도출 | 아키텍처 옵션은 사용자의 답변에서 합성한다. 미리 정해진 템플릿을 사용하지 않는다 |
| 구체적 정당화 | 추천에는 반드시 수집된 답변 항목을 인용하여 이유를 설명한다 |
| 순서 준수 | Step 1 → 2 → 3 → 4 → 5 → 6 순서를 반드시 지킨다 |
