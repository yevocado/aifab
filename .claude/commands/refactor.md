# /aifab:refactor

**담당 모델:** Advisor (Opus) + Sonnet (구현)
**선행:** /aifab:plan (리팩토링 Wave 정의)
**연계:** /aifab:execute (Wave 적용), /aifab:debug (회귀 발생 시)

---

## 목적

동작을 보존하면서 코드 구조를 개선한다. 기능 변경 없이 가독성, 성능, 구조를 향상시키는 안전한 리팩토링 워크플로우.

---

## 프로세스

### 1단계: 리팩토링 동기 명확화

다음 중 해당하는 동기를 식별한다.

- **가독성**: 네이밍 불명확, 중복 로직, 과도한 중첩
- **성능**: 불필요한 연산, 비효율적 자료구조, N+1 쿼리
- **구조**: 책임 분리 위반, 순환 의존성, 모듈 경계 불명확
- **Karpathy 원칙 위배**: 불필요한 추상화, 과도한 복잡성, 마법 같은 코드

동기가 불명확하면 진행을 중단하고 목적을 재정의한다.

### 2단계: 베이스라인 테스트 통과 확인

```
테스트 실행 → 전체 통과 확인
```

- 테스트가 없으면 **리팩토링 거부**
- `/aifab:plan`으로 돌아가 회귀 테스트를 먼저 작성한다
- 테스트 커버리지가 부족하면 scope 축소 또는 테스트 보강 후 진행

> 테스트 없는 리팩토링은 리팩토링이 아니라 도박이다.

### 3단계: 리팩토링 Scope 정의

명시적으로 다음을 문서화한다.

**영향 받는 파일 목록**
- 변경 대상 파일을 열거한다

**동작 보존 약속**
- 외부 인터페이스(API, 함수 시그니처, 반환값)는 동일하게 유지
- 부작용(파일 I/O, 네트워크, DB) 동작은 그대로

**변경하지 않을 영역**
- 이번 리팩토링에서 손대지 않을 파일/모듈을 명시
- scope 밖 코드는 절대 수정하지 않는다

### 4단계: 패턴 선택 (대규모 리팩토링 시)

**소규모** (파일 1-3개, 함수 단위): 직접 변경

**대규모** (모듈/레이어 수준): 다음 중 선택

- **Strangler-fig 패턴**: 새 구현을 병렬로 만들고 점진적으로 트래픽/호출을 전환
  - 적합: 외부 의존성이 많은 모듈 교체
- **Branch-by-abstraction 패턴**: 추상화 레이어 도입 후 구현체 교체
  - 적합: 내부 구현체 교체, 테스트 격리 필요 시

패턴을 선택한 이유를 한 줄로 기록한다.

### 5단계: 작은 단계로 분해

- 각 단계는 **5분 이내** 완료 가능한 크기로 분해
- 각 단계 후 테스트가 통과해야 다음 단계로 진행
- 단계 목록을 REFACTOR-LOG.md에 사전 기록

분해 기준:
- 하나의 함수/메서드 이동 또는 이름 변경
- 하나의 중복 제거
- 하나의 추상화 도입 또는 제거

### 6단계: 변경 → 테스트 → Commit (반복)

각 단계마다 다음 사이클을 반복한다.

```
1. 최소 변경 적용
2. 테스트 실행 (전체 또는 관련 suite)
3. 통과 확인
4. git commit (refactor: prefix, revertable 상태 유지)
```

commit 메시지 형식:
```
refactor: <변경 내용> (<영향 파일>)
```

테스트 실패 시:
- 즉시 변경을 되돌린다 (`git revert` 또는 수동 롤백)
- `/aifab:debug`로 원인 분석
- 단계를 더 작게 분해 후 재시도

### 7단계: 최종 검증

모든 단계 완료 후:

- [ ] 전체 테스트 suite 통과
- [ ] 성능 회귀 없음 (벤치마크 비교)
- [ ] 외부 동작 동일 (인터페이스 변경 없음)
- [ ] REFACTOR-LOG.md 완성
- [ ] 최종 보고서 작성 (이전/이후 메트릭)

---

## 금지 사항 (Anti-patterns)

- 테스트 없는 코드 리팩토링 (먼저 테스트 작성)
- "리팩토링 김에" 새 기능 추가
- 한 commit에 다수 변경 (각 변경 독립 commit)
- 마지막 빅뱅 머지 (점진 머지)

---

## Karpathy 원칙 적용

**수술적 변경**: 명시된 scope만 건드린다. scope 밖 코드에 손대는 순간 리팩토링이 아니라 개발이 된다.

**불필요한 추상화 추가 금지**: 리팩토링의 목표는 복잡성 감소다. 새 인터페이스, 새 레이어, 새 패턴을 추가하려는 충동을 경계한다.

**가정 없이 진행 금지**: "이 코드는 이렇게 동작할 것이다"는 가정은 금지. 테스트가 동작을 검증한다. 테스트가 없으면 가정도 없다.

## Output

### REFACTOR-LOG.md 생성/갱신

프로젝트 루트 또는 대상 모듈 디렉토리에 생성한다.

```markdown
# REFACTOR-LOG

## 리팩토링 개요
- **동기**: <동기>
- **날짜**: <날짜>
- **대상**: <파일 목록>

## 단계별 기록
| 단계 | 변경 내용 | 테스트 결과 | Commit |
|------|-----------|-------------|--------|
| 1    | ...       | 통과        | abc1234 |

## 동작 보존 증거
- 베이스라인 테스트: <통과 수>/<전체 수>
- 최종 테스트: <통과 수>/<전체 수>
```

### 각 단계별 git commit

```
refactor: <변경 내용>
```
