# `/aifab:uat` — 사용자 인수 테스트 (UAT) 가이드 및 결과 수집

**담당 모델:** Advisor (claude-opus-4-7) — 사용자와의 인터랙티브 진행이 필요한 작업

---

## 역할

Advisor로서 PLAN.md의 모든 기능에 대한 UAT 시나리오를 생성하고, 사용자와 인터랙티브하게 결과를 수집하며, 실패한 시나리오는 자동으로 수정 Wave로 전환한다.

---

## 사전 조건 확인

시작 전 다음 두 가지를 반드시 확인한다.

**1. Playwright 테스트 통과 여부 확인**

`WORKLOG.md`를 읽어 "Playwright: 통과" 항목이 있는지 확인한다.
- 존재하지 않으면 다음 메시지를 출력하고 즉시 중단한다:
  > "Playwright 테스트가 아직 통과되지 않았습니다. 먼저 `/aifab:playwright`를 실행해주세요."

**2. 앱 실행 여부 확인**

사용자에게 확인한다:
> "앱이 현재 실행 중이고 접근 가능한 상태인가요? (예/아니오)"

- "아니오"이면: "앱을 먼저 실행한 후 `/aifab:uat`를 다시 시작해주세요." 출력 후 중단한다.
- "예"이면: 1단계로 진행한다.

---

## 1단계: UAT 시나리오 생성

`PLAN.md`와 `ARCHITECTURE.md`를 읽어 각 Wave에 대한 UAT 시나리오를 생성한다.

**규칙:**
- Wave당 최소 1개 이상의 시나리오를 생성한다.
- 시나리오는 실제 사용자 관점에서 작성한다 (기술적 구현이 아닌 기능 동작 중심).
- 사전 조건, 단계, 기대 결과를 구체적으로 명시한다.

**시나리오 형식:**

```markdown
## UAT 시나리오 — <프로젝트명>
날짜: YYYY-MM-DD

### 시나리오 1: <기능명>
**목적:** 이 기능이 실제 사용 환경에서 올바르게 동작하는지 확인
**사전 조건:** <필요한 초기 상태 (예: 로그인 상태, 특정 데이터 존재 등)>
**테스트 단계:**
1. <구체적 단계>
2. <구체적 단계>
3. <구체적 단계>
**기대 결과:** <무엇이 화면에 보여야 하고 어떻게 동작해야 하는가>
**결과:** [ ] 통과  [ ] 실패  [ ] 부분 통과
**피드백:**
```

---

## 2단계: 시나리오 제시

생성한 전체 UAT 문서를 출력한다. 출력 후 다음 메시지를 표시한다:

> "아래 시나리오를 직접 테스트해주세요. 각 시나리오 완료 후 결과를 알려주세요. 준비되셨으면 '시작'이라고 입력해주세요."

사용자가 '시작' 또는 준비 완료 의사를 표현하면 3단계로 진행한다.

---

## 3단계: 결과 수집 (인터랙티브)

**각 시나리오마다 아래 절차를 반드시 순서대로 수행한다.**

시나리오 N에 대해 다음을 출력한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
시나리오 <N>: <제목>

<시나리오의 테스트 단계를 간략히 요약>

결과를 입력해주세요:
1. ✅ 통과
2. ❌ 실패
3. ⚠️ 부분 통과 (피드백 필요)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**중요: 사용자의 응답을 기다린다. 응답 없이 다음 시나리오로 넘어가지 않는다.**

- 사용자가 **1 (통과)**을 선택하면: "✅ 기록했습니다." 출력 후 다음 시나리오로 진행한다.
- 사용자가 **2 (실패)**를 선택하면:
  > "어떤 문제가 발생했나요? 오류 메시지나 예상과 다른 동작을 설명해주세요."
  사용자의 피드백을 받은 후 다음 시나리오로 진행한다.
- 사용자가 **3 (부분 통과)**를 선택하면:
  > "어떤 부분이 통과되지 않았나요? 구체적인 피드백을 알려주세요."
  사용자의 피드백을 받은 후 다음 시나리오로 진행한다.

모든 시나리오 결과를 수집하면 4단계로 진행한다.

---

## 4단계: 결과 처리

수집된 결과를 집계한다.

### 모든 시나리오가 통과된 경우

1. `WORKLOG.md`에 다음 내용을 추가한다:
   ```markdown
   ## [YYYY-MM-DD] UAT 결과
   - UAT: 전체 통과 (YYYY-MM-DD)
   - 총 시나리오: N개 | 통과: N개 | 실패: 0개
   ```

2. Git 태그를 생성한다:
   ```bash
   git tag -a v1.0.0 -m "UAT 통과 — 프로덕션 준비 완료"
   ```

3. 다음 메시지를 출력한다:
   > "🎉 UAT 완료! 프로덕션 배포 준비가 되었습니다. 5단계 최종 서명 절차를 진행합니다."

4. 5단계로 진행한다.

---

### 실패 또는 부분 통과 시나리오가 있는 경우

1. 모든 실패/부분 통과 시나리오와 피드백을 정리한다.

2. `PLAN.md`에 새 수정 Wave를 추가한다:

   ```markdown
   ### Wave N+1: UAT 피드백 수정 [Small/Medium]
   **목표:** UAT 실패 시나리오 수정
   **실패 시나리오:**
   - 시나리오 <번호>: <제목> — <실패 내용 요약>
   **사용자 피드백:** <수집된 피드백>
   **완료 기준:**
   - [ ] 실패했던 UAT 시나리오 재테스트 통과
   - [ ] 부분 통과 항목 전체 통과로 전환
   ```

   Wave 크기는 실패 범위와 복잡도에 따라 Small 또는 Medium으로 결정한다.

3. `WORKLOG.md`에 다음 내용을 추가한다:
   ```markdown
   ## [YYYY-MM-DD] UAT 결과
   - UAT: 부분 통과 (YYYY-MM-DD)
   - 총 시나리오: N개 | 통과: N개 | 실패/부분통과: N개
   - 실패 시나리오: <목록>
   - 수정 Wave Wave N+1 추가됨
   ```

4. 다음 메시지를 출력한다:
   > "UAT 피드백을 반영하여 Wave N+1 수정 Wave를 PLAN.md에 추가했습니다.
   > `/aifab:execute`로 수정을 시작하세요.
   > 수정 완료 후 `/aifab:playwright`를 실행하고, 이후 `/aifab:uat`로 실패 시나리오만 재테스트합니다."

**재테스트 시:** `/aifab:uat`가 다시 실행되면 이전에 실패한 시나리오만 대상으로 3단계를 진행한다. 이미 통과된 시나리오는 건너뛴다.

---

## 5단계: 최종 서명

모든 UAT가 통과(초기 통과 또는 재테스트 통과 포함)된 후 진행한다.

1. `docs/` 디렉토리가 없으면 생성하고, `docs/UAT-REPORT.md`를 아래 형식으로 작성한다:

   ```markdown
   # UAT 완료 보고서
   프로젝트: <이름> | 날짜: YYYY-MM-DD
   총 시나리오: N | 통과: N | 실패: 0

   ## 시나리오별 결과

   | # | 시나리오 | Wave | 결과 | 비고 |
   |---|----------|------|------|------|
   | 1 | <제목> | Wave 1 | ✅ 통과 | |
   | 2 | <제목> | Wave 2 | ✅ 통과 | 재테스트 통과 |

   ## UAT 이력
   - 1차 테스트: YYYY-MM-DD — N개 통과, N개 실패
   - 수정 후 재테스트: YYYY-MM-DD — 전체 통과

   ## 최종 의견
   <프로젝트명>의 모든 기능이 사용자 인수 테스트를 통과하였습니다.
   프로덕션 배포 준비가 완료되었습니다.
   ```

2. `WORKLOG.md`에 다음 내용을 추가한다:
   ```markdown
   ## [YYYY-MM-DD] UAT 최종 완료
   - docs/UAT-REPORT.md 생성
   - 전체 시나리오 통과 확인
   ```

3. 최종 Git 커밋을 생성한다:
   ```bash
   git add docs/UAT-REPORT.md WORKLOG.md PLAN.md
   git commit -m "chore: UAT passed - project complete"
   ```

4. 다음 메시지를 출력한다:
   > "🎉 프로젝트 완료! UAT 보고서가 docs/UAT-REPORT.md에 저장되었습니다.
   > 모든 기능이 검증되었으며 프로덕션 배포 준비가 완료되었습니다."

---

## 주의사항

- **인터랙티브 수집의 핵심:** 시나리오 결과를 한꺼번에 묻지 않는다. 반드시 한 시나리오씩 순서대로 진행한다.
- **재테스트 범위:** 재테스트 시에는 이전에 실패한 시나리오만 진행한다. 통과된 시나리오는 재테스트하지 않는다.
- **피드백 보존:** 사용자의 피드백은 수정 Wave 작성 시 정확히 반영한다. 요약하거나 생략하지 않는다.
- **Wave 크기 판단:** 수정 Wave의 크기는 실패 시나리오의 수와 복잡도를 함께 고려하여 결정한다 (1~2개 단순 수정 = Small, 3개 이상 또는 복잡한 수정 = Medium).
- **Git 태그 조건:** `v1.0.0` 태그는 초기 전체 통과 시에만 생성한다. 재테스트 통과 시에는 태그를 생성하지 않는다.
